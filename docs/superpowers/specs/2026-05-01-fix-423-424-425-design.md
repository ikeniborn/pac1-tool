## FIX-423 — Contract gate matching

### Проблема

`agent/evaluator.py:~450` проверяет, является ли строка `required_evidence` подстрокой
`refs_str` (склеенные пути grounding_refs). Но evaluator LLM генерирует дескриптивные
строки вида:

```
'Final listing of /02_distill/cards/ showing empty'
'Read of account file showing primary_contact_id for Blue Harbor Bank'
```

Эти строки никогда не являются подстрокой голого пути `/02_distill/cards/`. Результат:
каждый нетривиальный контракт — 2 reject → агент сдаётся → NONE_CLARIFICATION или
DENIED вместо OUTCOME_OK.

Подтверждено в логах:
- t01: `[evaluator] REJECTED (1/2): Contract required_evidence missing...`
- t17, t26, t35: аналогично

### Фикс

Файл: `agent/evaluator.py`, строки 446–455.

**Инвертировать проверку**: считать evidence покрытым если хотя бы один grounding_ref
является подстрокой строки evidence (т.е. путь упоминается в описании).

```python
# Было:
refs_str = "\n".join(refs).lower()
missing = [e for e in contract.required_evidence if e.lower() not in refs_str]

# Станет:
missing = [
    e for e in contract.required_evidence
    if not any(ref.lower() in e.lower() for ref in refs)
]
```

### Корректность

- Грейдинг: если путь `/accounts/acct_001.json` присутствует в evidence-описании
  `'Read of /accounts/acct_001.json showing...'` — матч. Агент прочитал файл.
- Если агент не читал файл — пути нет ни в одном ref → evidence остаётся в `missing` → reject.
- Нет ложных срабатываний: грейдинг строже (требует буквального пути из refs
  в тексте evidence), но не мягче.

## FIX-424 — Lookup evaluator bypass

### Проблема

`agent/loop.py:81–97`, функция `_should_bypass_evaluator_lookup()`:

```python
def _should_bypass_evaluator_lookup(report) -> bool:
    if report.outcome == "OUTCOME_NONE_CLARIFICATION":
        return True
    _steps = report.completed_steps_laconic or []
    _explored = any(
        any(kw in s.lower() for kw in ("list", "read", "search", "find", "tree"))
        for s in _steps
    )
    return _explored  # bypass если было exploration
```

t30 (2 шага, неправильный ответ) и t40 (12 stall-эскалаций, неправильный ответ):
оба lookup + OUTCOME_OK + есть хотя бы один read → evaluator не запускался → неверный
ответ прошёл без проверки.

### Фикс

Файл: `agent/loop.py`, функция `_should_bypass_evaluator_lookup`.

Bypass только для NONE_CLARIFICATION:

```python
def _should_bypass_evaluator_lookup(report) -> bool:
    """FIX-424: bypass evaluator for lookup only on NONE_CLARIFICATION."""
    return report.outcome == "OUTCOME_NONE_CLARIFICATION"
```

### Корректность

- Исходная цель FIX-420: не блокировать корректные refusals. Сохраняется.
- OUTCOME_OK теперь всегда проходит через evaluator — нужное поведение.
- Нет риска регрессии: evaluator для lookup с OUTCOME_OK ранее всегда был `mid`
  skepticism — не более строгим чем для других типов.
- Сохранить оригинальный docstring функции, обновив только тело и FIX-номер.

## FIX-425 — CRM date verification gate

### Проблема

t13: VAULT_DATE=2026-07-25, агент вычислил due_on=2026-08-08 (14 дней). CRM-правило
требует TOTAL_DAYS = stated_days + 8. Без +8 offset результат неправильный.
Evaluator (`mid` skepticism) не проверяет арифметику детально.

### Фикс

Файл: `agent/loop.py`, добавить hard-gate до evaluator (~строка 2233).

Перед тем как позволить CRM OUTCOME_OK пройти на evaluator, проверить что в
`completed_steps_laconic` агент явно указал VAULT_DATE и CRM-смещение:

```python
# FIX-425: CRM date anchor gate — before evaluator
if (task_type == TASK_CRM
        and isinstance(job.function, ReportTaskCompletion)
        and job.function.outcome == "OUTCOME_OK"):
    _steps_str = " ".join(job.function.completed_steps_laconic or []).lower()
    _has_vault = any(k in _steps_str for k in ("vault_date", "vault-date", "vault date"))
    _has_offset = any(k in _steps_str for k in ("+8", "total_days", "crm offset", "8 day"))
    if not _has_vault or not _has_offset:
        st.eval_rejections += 1
        _hint = (
            "[crm-gate] FIX-425: CRM date was computed without explicit VAULT_DATE anchor "
            "or +8-day CRM offset. "
            "State in completed_steps: 'VAULT_DATE=X, stated=N days, "
            "TOTAL_DAYS=N+8=M, due_on=VAULT_DATE+M=YYYY-MM-DD'."
        )
        print(f"{CLI_RED}[crm-gate] FIX-425: missing VAULT_DATE or +8 offset in steps{CLI_CLR}")
        st.log.append({"role": "user", "content": _hint})
        return _hint
```

Возврат строки прерывает dispatch (аналогично другим pre-checks в loop.py).

### Риски и митигация

- **False reject**: агент вычислил верно, но не упомянул "vault_date" в steps.
  Митигация: промпт уже требует `"VAULT_DATE=X ... BASE±N=Z"` в `current_state`.
  Добавить упоминание в CRM-промпт: требовать строку с "+8" или "TOTAL_DAYS" в
  `completed_steps_laconic`.
- **Scope**: gate срабатывает только при `task_type == TASK_CRM` и `OUTCOME_OK`.
  Не затрагивает другие типы.
