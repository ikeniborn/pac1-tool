# Дизайн: Структурированный аудит выполнения t43

**Дата:** 2026-04-30  
**Цель:** Запустить PAC1-задачу t43 и оценить эффективность текущей мультиагентной архитектуры с контрактами и субагентами, а также качество вики и графа знаний.

---

## Контекст

Текущая архитектура (Hub-and-Spoke) реализована в рамках плана `2026-04-30-multi-agent-architecture.md`. Агент разбит на 9 специализированных субагентов, общающихся через Pydantic-контракты. Граф знаний содержит **675 узлов** и **1063 рёбра**. Накоплено **80 DSPy-примеров**.

---

## Методология: Структурированный аудит (Подход B)

### Фаза 1 — Baseline (до запуска)

Зафиксировать:
- Количество узлов/рёбер в `data/wiki/graph.json`
- Состояние wiki-страниц (качество, количество фрагментов) по типу задачи t43
- Активные env-переменные: `CONTRACT_ENABLED`, `WIKI_GRAPH_ENABLED`, `WIKI_GRAPH_FEEDBACK`

### Фаза 2 — Запуск с диагностикой

```bash
LOG_LEVEL=DEBUG make task TASKS=t43
```

Что наблюдать в реальном времени:
- Какой тип задачи определил ClassifierAgent
- Прошла ли фаза контрактной переговорки (CONTRACT_ENABLED=1)
- Сколько раундов переговоров потребовалось
- Были ли blocking_objections или consensus достигнут сразу
- Сколько шагов выполнил агент
- Были ли stall-детекции

### Фаза 3 — Анализ по 4 измерениям

#### 3.1 Эффективность контрактов
| Метрика | Что смотрим |
|---|---|
| Фаза CONTRACT | Успешно завершена / упала в fallback |
| Раунды переговоров | ≤3 (CONTRACT_MAX_ROUNDS=3) |
| Blocking objections | Были ли блокирующие возражения |
| Консенсус | Достигнут с первого раунда или после корректировок |
| Влияние на план | Изменил ли контракт финальный промпт (addendum) |

#### 3.2 Координация субагентов
| Субагент | Проверяем |
|---|---|
| ClassifierAgent | Корректный task_type, confidence |
| PlannerAgent | Правильно собрал addendum из контракта + wiki |
| WikiGraphAgent | Количество инжектированных узлов (injected_node_ids) |
| VerifierAgent (evaluator) | Запустился ли перед report_completion, что оценил |
| SecurityAgent | Были ли отклонения по скоупу |
| StallAgent | Было ли обнаружение стоп-петель |

#### 3.3 Качество Wiki и Графа
| Компонент | Метрика |
|---|---|
| Wiki-страница | quality уровень, количество фрагментов для типа задачи |
| Graph injection | top-K релевантных узлов попали в промпт |
| Post-trial feedback | Изменился ли граф после прогона (bump_uses / degrade) |
| Wiki-lint | Добавлен ли новый фрагмент в страницу |
| Graph autobuild | Добавлены ли новые узлы из результата задачи |

#### 3.4 Итоговое качество решения
| Метрика | Что смотрим |
|---|---|
| Score задачи | 0.0 / 0.5 / 1.0 |
| Количество шагов | <10 (эффективно) / 10-20 (нормально) / 20-30 (много) |
| Инструменты | Какие PCM-инструменты вызывались |
| Stall-детекции | 0 = хорошо |
| Write-scope нарушения | 0 = хорошо |

### Фаза 4 — Итоговая оценка

По каждому измерению вывести:
- Что работает хорошо
- Что работает плохо или неэффективно
- Конкретные рекомендации по улучшению

---

## Критерии успеха

Оценка считается завершённой когда:
1. t43 выполнена (score получен)
2. Все 4 измерения проанализированы с конкретными данными из логов
3. Граф и wiki проверены до/после
4. Сформулированы минимум 3 конкретные рекомендации
