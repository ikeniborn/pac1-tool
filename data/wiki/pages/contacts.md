## Key pitfalls
**Stale contact data risk**: Multiple contacts show `last_seen_on` dates far in the past (some as early as <date>), suggesting stale records where role, email, or employment status may have changed. This can lead to sending communications to wrong people or wrong roles.

**Data conflict from concurrent reads**: When reading the same contact or manager file at different times, the agent received different data:
- `/contacts/<file> returned Caroline Lehmann (Finance Director) at one timestamp and Sandra Sauer (Head of Engineering) at another
- `/contacts/<file> returned Liv van Leeuwen (QA Lead) and later Indy van Wijk (Product Manager)
- `/contacts/<file> returned Johannes Krüger (Finance Director) and later Jana Lorenz (Operations Director)
- `/contacts/<file> returned three different people: Ines Möller, Christoph Adler, and Tim Hoffmann

This pattern suggests the underlying data store was being modified during read operations, making deterministic behavior impossible without locking or versioning mechanisms.

## Shortcuts
**File Path Patterning:**
- All contacts consistently located under `/contacts/<file> directory
- File IDs follow two distinct schemas: `mgr_XXX` (managers) and `cont_XXX` (regular contacts)
- Path-based indexing is reliable; no need to scan parent directories

**Caching Opportunity:**
- `<contact>.json` read across 3 separate tasks (t14, t37, t38)
- `<manager>.json` read across 2 separate tasks (t36, t39)
- High probability of repeated reads within same session suggests strong benefit from session-level contact caching
- Same-file reads (t12/t18 for <contact>, t12/t19 for <contact>, t12/t38 for <contact>) returned different records entirely, making standard content caching unreliable without additional context parameters

**Data Consistency Concern:**
- `<contact>.json` returned different full_name values across reads: "Sem Bakker" (t14), "Isa Meijer" (t37), "Milan de Boer" (t38)
- `<manager>.json` similarly returned "Martin Herzog" (t36) and "Hannah Hartmann" (t39)
- Same contact file returning different records indicates file may contain multiple entries indexed by account, or file is dynamically aggregated
- **New evidence of multi-record files:** <manager>.json returned "Ines Möller" (t16), "Christoph Adler" (t37), and "Tim Hoffmann" (t39); <contact>.json returned "Caroline Lehmann" (t12) and "Sandra Sauer" (t18); <contact>.json returned "Liv van Leeuwen" (t12) and "Indy van Wijk" (t19); <contact>.json returned "Johannes Krüger" (t12) and "Jana Lorenz" (t38)
- **Optimization**: Validate file structure assumptions before bulk-read operations; do not assume single-record-per-file
- **Critical**: When re-reading a file, do not assume the previously cached record is still valid; treat each read as potentially returning a different record based on task context

**Email Domain Clustering:**
- Contacts within same account share email domain (e.g., blue-harbor-bank.example.com has 3 contacts: <contact>, Isa Meijer, Milan de Boer)
- Pre-fetching related contacts by account_id may reduce read calls
- Same-domain pattern confirmed: acme-robotics.example.com (<contact>, <contact>), blue-harbor-bank.example.com (<contact>, <contact>), silverline-retail.example.com (<contact>, <contact>)

**Contact Metadata Patterns:**
- `preferred_tone` field varies: "direct", "warm", "brief", "formal"
- Tone preference should be read and applied when generating communication drafts
- `last_seen_on` dates can be stale (oldest: <date>, newest: <date>); do not rely on this for real-time presence indicators

**Record-Context Alignment:**
- When a contact file returns a different record on re-read, always verify the returned record's `account_id` matches the current task's context before using the data
- Multiple records per file suggests implicit filtering on account context; mismatched records should be treated as stale data for the current task
