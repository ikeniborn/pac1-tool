## Workflow steps
**Vault Anchor Derivation**
- Simple temporal references ("tomorrow", "yesterday", "next week", "in 2 weeks") resolve directly from system date without ambiguity
- Complex relative anchors ("43 days ago", "the article I captured") fail when the anchor point is external or vague
- Establish a firm vault anchor before attempting candidate file inversion
- Simple date arithmetic (OUTCOME_OK) is reliable; anchor-based derivations risk OUTCOME_NONE_CLARIFICATION
- Explicit output format specifications (e.g., "DD-MM-YYYY only") should be captured as part of task intent, not inferred
- Explicit date format requirements ("Answer only YYYY-MM-DD") can introduce OUTCOME_NONE_CLARIFICATION if not properly captured as part of task intent; format constraints are distinct from date resolution and must be handled separately

**Candidate File Inversion Approach**
- When task references a vague temporal anchor, first attempt to derive the absolute date
- If derivation yields ambiguous or unresolvable reference, surface ambiguity to user for clarification rather than proceeding blind
- Date-relative queries with explicit durations ("43 days ago") succeed when anchor is system date; fail when anchor requires external context
- Invert candidate files only after confirmed absolute date anchor
- Direct simple temporal references ("day before yesterday") resolve from system date without external context; failure to produce OUTCOME_OK in these cases indicates a separate execution issue rather than anchor derivation failure
- Queries combining temporal anchors with task-specific context ("the article I captured 6 days ago") can succeed when the duration is explicit and anchor is system date, even with attached context; treat these as OUTCOME_OK candidates when date arithmetic is clean
- When OUTCOME_NONE_CLARIFICATION occurs on what should be simple date arithmetic, verify whether format specification or output constraint handling is the blocking factor rather than date resolution
- Explicit format requirements ("Answer only YYYY-MM-DD") do not inherently block OUTCOME_OK when both temporal calculation (system date anchor, clean arithmetic) and format specification are straightforward and properly captured as part of task intent; t41 confirms simple forward-duration queries with clear format output can achieve OUTCOME_OK despite format constraints
- Forward-duration queries ("in X days") using system date as anchor show clean arithmetic properties similar to backward-duration queries; both succeed with explicit durations when anchor is system date; t41 demonstrates forward query with explicit format ("DD-MM-YYYY only") achieved OUTCOME_OK, confirming direction of duration (past vs future) does not inherently impact resolution when anchor is clear
- The distinction between forward and backward durations matters for error diagnosis: backward queries that fail suggest anchor ambiguity, while forward queries that fail more likely indicate execution or format handling issues rather than anchor derivation problems
- Even with clean date arithmetic (system date anchor, explicit duration) and straightforward temporal resolution, queries asking to identify specific files ("which one was it?") may still result in OUTCOME_NONE_CLARIFICATION if the file itself cannot be definitively matched or located; the failure point shifts from date resolution to file identification when date calculation is clean but no matching candidate exists
- The critical distinction between context-as-modifier versus context-as-subject: queries using task-specific references as descriptive context (e.g., referencing an article to set temporal scope) can succeed, while queries where the task-specific reference IS the subject of identification ("which article did I capture?") fail when that reference cannot be resolved independent of the query; t42 confirms this by failing on "which article" identification even with clean backward arithmetic (13 days ago, system date anchor), because the external context provides no candidate to invert

## Key pitfalls
Temporal Anchor Mismatch

The system maintains two distinct time references: system-clock (actual wall-clock time) and vault-time (the temporal context of the current task/operation). When these diverge, the agent may anchor calculations to the wrong reference point.

**Risk**: If the agent uses system-clock instead of vault-time, temporal arithmetic (e.g., "2 days from now") will be computed against the wrong baseline. For task t41, vault-time was <date>, but if the agent referenced system-clock, it would calculate against today's actual date rather than the vault's perceived "now".

**Risk**: Wrong temporal anchor. The agent may implicitly adopt the wrong "now" when processing temporal queries. For date arithmetic, it must use the vault's temporal context as the anchor, not infer from system time. In t41, the query "What date is in 5 days?" required the agent to recognize that "now" meant <date> (vault-time), not the system's actual current date.

**Risk**: Premature NONE_CLARIFICATION. If the agent resolves a temporal query without verifying it used the correct temporal anchor (vault-time vs. system-clock), it may provide a plausible but incorrect date. This is especially dangerous in list/find/tree operations that filter by date ranges—using the wrong anchor produces silent data mismatches rather than obvious errors. The t41 task type (temporal) with a simple "What date is in 5 days?" prompt illustrates how even straightforward queries require explicit anchor verification to avoid confusion. Additionally, when temporal arithmetic precedes a retrieval operation (e.g., "article I captured 43 days ago"), a NONE_CLARIFICATION outcome may stem from ambiguity about whether the anchor calculation failed, no data exists for the calculated date, or clarification is genuinely needed—a distinction that requires checking whether data exists at the computed timestamp before marking NONE_CLARIFICATION.

## Temporal Patterns
### VAULT_DATE as Lower Bound
- The vault contains files dated `<date>` indicating this is the most recent known temporal anchor
- VAULT_DATE establishes the lower bound for temporal calculations—assumed to be the agent's best estimate of "today"

### ESTIMATED_TODAY Derivation
- When explicit "today" is unavailable, ESTIMATED_TODAY may be derived from the most recent file modification date
- Example: If the newest file in the vault is dated `<date>`, ESTIMATED_TODAY is inferred as `<date>`

### Candidate Inversion (implied_today = file_date + N)
- When the task asks for a relative date (e.g., "what day is in 2 days?"), the agent must:
  1. Establish the reference date (VAULT_DATE or ESTIMATED_TODAY)
  2. Add the specified offset (N)
  3. Return the computed date in required format (DD-MM-YYYY)
