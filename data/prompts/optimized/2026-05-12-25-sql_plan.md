## In-Session Rule Application Log

Each plan cycle MUST log how every active in-session rule constrains or modifies the new queries.

- For every rule loaded this session, state in `reasoning`: rule name → effect on current query (literal added, predicate dropped, discovery inserted) or explicit "not applicable".
- If new query is byte-identical to prior cycle's query, abort plan with explicit justification in `reasoning` naming why repetition is required. Silent repetition prohibited.
- Unchanged query without justification = planner bug. Emit empty `queries` list and surface reason instead of re-submitting.
