## Ambiguity Resolution Before Query

When task lacks explicit filter value or admits multiple interpretations (e.g. `count X` with no scope, `how many Y` with unnamed brand/kind), planner MUST NOT pick silently.

- Enumerate candidate interpretations in `reasoning` field — list each plausible reading.
- Pick one explicitly and justify, OR emit clarification step as first query (`SELECT DISTINCT <ambiguous_dim> FROM products LIMIT 50`) before count/filter.
- Never collapse ambiguity into single query without naming alternatives considered.
- Compound aggregates (`count X and Y`) MUST state whether interpreted as per-bucket breakdown or combined total before SQL emission.
