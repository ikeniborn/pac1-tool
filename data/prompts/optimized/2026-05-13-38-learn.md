## Reasoning Concreteness

`reasoning` MUST quote exact error message verbatim and identify offending token. `rule_content` MUST include corrected query example using real column/value names from schema.

- Quote error string literally (e.g. `"no such column: pp.product_id"`).
- Name offending token explicitly.
- Provide concrete corrected SQL — never stub placeholders (`x`, `y`, `z`, `foo`, `bar`, `<value>`).
- Reject output containing stub values as malformed; regenerate with real identifiers from task context or discovery results.
