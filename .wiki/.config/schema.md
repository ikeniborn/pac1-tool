# Конвенции написания wiki-страниц

## Язык и стиль

### Основные правила
- **Основной язык**: русский
- **Технические термины** (SQL, API, LLM, ETL, SCD, TTL, DDL, JSON, YAML) — не переводить
- **Имена систем**: оригинальное написание — RT.DataExporter, CRM B2C, ЦХД, СИП
- **Аббревиатуры**: расшифровывать при первом использовании на странице
  - Пример: "ЦХД (Центральное хранилище данных)"
- **Стиль**: нейтральный, информативный, без разговорных оборотов и оценочных суждений

### Недопустимые формулировки
- "Очевидно, что..." / "Понятно, что..."
- "Лучший способ..." (без атрибуции источника)
- "Кажется, что..." / "По всей видимости..."
- Местоимения "я", "мы", "наш"

## Именование файлов и папок

### Файлы wiki-страниц
- Формат: kebab-case, кириллица допустима
- Примеры корректных имён:
  - `версионирование-scd.md`
  - `rt-data-exporter.md`
  - `clickhouse-обзор.md`
  - `поток-gp-s3-ch.md`
- Запрещено: пробелы, точки (кроме расширения), спецсимволы кроме дефиса

### Папки доменов
- Нижний регистр, латиница (для надёжности WikiLink на case-sensitive FS)
- `ии/`, `ростелеком/`, `базы-данных/`
- Подпапки аналогично: `архитектура-данных/`, `системная-архитектура/`

### Заголовки страниц (H1)
- Русское название концепции или системы
- Технический термин в скобках при необходимости: `# Версионирование (SCD1/SCD2)`
- Для имён систем: `# RT.DataExporter`

## Структура wiki-страницы

### Обязательные элементы (в порядке следования)

**1. Frontmatter (YAML)**
```yaml
---
wiki_sources: ["[[пути к источникам]]"]
wiki_updated: YYYY-MM-DD
wiki_status: stub | developing | mature
wiki_outgoing_links: ["[[связанная-страница]]", "[[другая-страница|отображаемое имя]]"]
wiki_external_links: ["https://example.com/docs"]
tags: [из tag-hierarchy.json]
aliases: [альтернативные названия]
---
```

**2. Заголовок H1**
```markdown
# Название сущности
```

**3. Вводный абзац (определение)**
Сразу после H1, без заголовка раздела. 1-3 предложения — что это такое, зачем нужно.

**4. Раздел "Основные характеристики" (H2)**
```markdown
## Основные характеристики
```
Ключевые свойства и параметры сущности.

**5. Раздел "Связанные концепции" (H2)** — опциональный

Используется только если нужно добавить пояснительный контекст к связям. Все WikiLinks для графа должны быть в `wiki_outgoing_links` frontmatter.

```markdown
## Связанные концепции

- [[другая-страница-вики]] — используется как основа для X
- [[ещё-одна-страница|отображаемое имя]] — альтернативный подход
```

Без описательного контекста раздел не создавать — использовать только `wiki_outgoing_links`.

### Опциональные разделы (добавлять при наличии информации)

- `## Применение в контексте [Домен]` — специфика применения
- `## Примеры` — конкретные примеры из источников
- `## Ограничения` — известные ограничения
- `## Best Practices` — рекомендации из источников
- `## История изменений` — если сущность эволюционировала (версии, даты)

## WikiLinks

### Когда использовать
- Ссылаться на другие wiki-страницы через `[[имя-страницы]]`
- При несовпадении имени файла с нужным текстом: `[[имя-файла|отображаемый текст]]`

### Запрещено
- Создавать "мёртвые" ссылки — только на уже существующие страницы вики
  - Исключение: можно создать ссылку с пометкой `<!-- TODO: создать страницу -->` если сущность явно важна
- Ссылаться на источники через WikiLinks — только через `wiki_sources` в frontmatter
- Ссылаться на файлы вне `.wiki/` через WikiLinks (только через `wiki_sources`)

### Как проверить существование страницы
Перед созданием WikiLink — проверить наличие файла в `index.md`.

## Правила работы с контентом

### Синтез, не копирование
- Информацию из источников переработать и структурировать
- Дословные цитаты допустимы только в code-блоках (SQL, конфигурации)
- Если нужно сохранить формулировку — указать источник явно:
  > По документу [RT.DataExporter HLD](wiki_sources): "..."

### Атрибуция фактов
При добавлении информации из нового источника указывать дату и источник:
```markdown
## История изменений

- **2026-04-14** (из [[!Daily/2026-04-14 Встреча по СИП]]): принято решение о переходе на ClickHouse v24.8
```

### Таблицы
Использовать для структурированных данных (параметры, сравнения, матрицы).
Markdown-таблицы с выравниванием:
```markdown
| Параметр | Значение | Примечание |
|----------|----------|-----------|
| Retention hot | 2 года | SLA для аналитики |
```

### Кодовые блоки
Всегда указывать язык:
````markdown
```sql
SELECT ...
```

```yaml
key: value
```
````

## Frontmatter — правила полей

### wiki_sources
- Массив реальных путей относительно корня репозитория
- Только файлы, которые реально были прочитаны
- Добавлять новые пути при каждом UPDATE, не удалять старые
- Пример: `["[[vaults/Work/Ростелеком/Системная архитектура/HLD/СИП.md]]"]`

### wiki_status
| Значение | Критерии |
|----------|---------|
| `stub` | Страница создана, < 2 источников, < 10 предложений контента |
| `developing` | ≥ 2 источника, ≥ 10 предложений, основные разделы заполнены |
| `mature` | ≥ 4 источника, все ключевые разделы, стабильный контент |

### tags
Использовать только теги из `knowledge-manager/shared/tag-hierarchy.json`.
Основные теги по доменам:
- ИИ: `ai`, `ai/agent`, `ai/claude-code`, `ai/llm`
- Ростелеком: `client/rostelecom`, `architecture/hld`, `architecture/dmd`
- БД: `database`, `database/name/clickhouse`, `database/name/postgresql`

### aliases
Альтернативные имена, по которым пользователь может искать эту страницу.
Включать: аббревиатуры, английские варианты, синонимы.
Пример: `aliases: ["SCD", "Slowly Changing Dimensions", "медленно меняющиеся измерения"]`

### wiki_outgoing_links
Массив WikiLinks на другие страницы вики — **основной механизм формирования графа Obsidian**.

Правила:
- Синтаксис элементов: `"[[имя-файла]]"` или `"[[имя-файла|отображаемое имя]]"`
- Только ссылки на существующие страницы (проверять по `index.md`)
- Obsidian подхватывает `[[]]`-синтаксис из frontmatter и строит рёбра графа
- Добавлять при каждом `ingest`/`update`: новые связанные сущности → в `wiki_outgoing_links`

Пример:
```yaml
wiki_outgoing_links:
  - "[[clickhouse-обзор]]"
  - "[[версионирование-scd|SCD]]"
```

### wiki_external_links
Массив внешних URL — справочные ресурсы, документация, тикеты. Не формируют граф.

Правила:
- Только `http://` или `https://` URL
- Добавлять ссылки на официальную документацию, HLD-документы вне репозитория, тикеты

Пример:
```yaml
wiki_external_links:
  - "https://clickhouse.com/docs/ru/engines/table-engines/mergetree-family/replacingmergetree"
```

## Запрещено на wiki-страницах

1. **Placeholder-текст**: "TODO", "см. источник", "будет добавлено"
2. **Мнения без атрибуции**: "лучший подход — ..." (без указания источника)
3. **Дублирование страниц**: перед созданием проверять `index.md`
4. **Удаление существующей информации**: только добавление и обновление
5. **Пустые разделы**: если раздел нечем заполнить — не создавать его
