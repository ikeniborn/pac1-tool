# Temporal gap formula — неверный ESTIMATED_TODAY

**Статус:** не решена (FIX-357 существует, но проблема остаётся)  
**Влияние:** t41 — 0/25 (0%), t42 — 0/25 (0%) во всех версиях v1–v4

## Симптом

Агент получает VAULT_DATE, добавляет фиксированный offset (+5 дней) и получает ESTIMATED_TODAY. Этот ESTIMATED_TODAY используется для вычисления дат в temporal задачах. Вычисление стабильно неверно.

- t41: "Какая дата через N дней?" → агент берёт VAULT_DATE + 5 + N → неверно
- t42: "Статья N дней назад" → агент вычисляет неверную дату → не находит файл → CLARIFICATION

## Почему накопленные знания не помогают

Это **баг в коде** (FIX-357 логика), не в промпте. Граф накопил правило "expand search ±1-2 days" — агент читает его, но базовая дата всё равно неверная, ±2 дня не помогают при ошибке в 5+ дней.

Vault перегенерируется каждый прогон: VAULT_DATE меняется, N меняется — нет одной фиксированной комбинации для накопления.

## Архитектурная проблема

Нет единого надёжного источника "сегодняшней даты" для агента. Агент пытается вывести её из содержимого vault, что нестабильно при рандомизации.

## Рекомендации

1. Не добавлять фиксированный offset к VAULT_DATE — использовать VAULT_DATE напрямую или проверять несколько источников.
2. Добавить явное поле `CURRENT_DATE` в AGENTS.MD vault-файл.
3. Если VAULT_DATE не найден — запрашивать дату явно, а не угадывать.
4. Рассмотреть injecting сегодняшней даты в system prompt напрямую из `currentDate`.
