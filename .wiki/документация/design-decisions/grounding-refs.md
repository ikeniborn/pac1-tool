# Grounding Refs (ссылки на позиции каталога)

Механизм верификации ответов агента: поле `grounding_refs` в финальном JSON связывает каждый ответ с конкретными SKU из каталога, предотвращая галлюцинации и обеспечивая трассируемость. Источник значений — AUTO_REFS, инжектируемые в сообщение задачи автоматически.

## Основные характеристики

- **Источник:** только AUTO_REFS-значения из сообщения задачи; не допускается ручная конструкция путей
- **Формат:** массив путей вида `/proc/catalog/SKU.json`
- **Связь с outcome:** пустой массив разрешён только при нулевых результатах запроса или агрегатах без `path`-колонки

## Запрещённые источники

| Источник | Причина запрета |
|----------|----------------|
| Пути, сконструированные из `sku` вручную (`/proc/catalog/{sku}.json`) | Создаёт риск несуществующих путей |
| Сырая колонка `path` из SQL без AUTO_REFS | Не прошла нормализацию AUTO_REFS |
| Изобретённые или угаданные пути | Отсутствие реального соответствия в каталоге |
| Значения из агрегатных запросов (`COUNT`, `SUM`, `AVG`) | Не содержат колонку `path` |

## Правила по типу ответа

| Тип ответа | Требование к grounding_refs |
|-----------|----------------------------|
| YES / товар найден | ≥1 SKU из SQL-результатов |
| COUNT / агрегат | ≥1 пример SKU из базовых строк |
| Нулевой счётчик | Пустой массив допустим |
| Нет колонки `path` в SQL | Обязательно `[]`; в `message` — описать, что подтверждено discovery |
| Пустой массив при обязательном числовом ответе | Триггер `OUTCOME_NEED_MORE_DATA` и LEARN-цикла |

## Различие: discovery hit vs SKU hit

Ключевое различие, которое обязательно отражать в `message`:
- **Discovery hit** — ключ присутствует в каталоге бренда (discovery-запрос). Означает лишь наличие ключа где-либо в каталоге.
- **SKU hit** — конкретный SKU содержит комбинацию ключ+значение (требует финального filter-запроса по набору SKU).

Discovery hit ≠ SKU hit. Перед утверждением о соответствии SKU — обязательно выполнить filter-запрос.
